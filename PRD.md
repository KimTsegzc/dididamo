任务：完成一款两轮摩托车打车平台BS网页端产品原型设计，对标滴滴出行核心业务逻辑，运力载体为摩托车而非四轮汽车，采用分阶段迭代规划；当前为演示展示阶段，**暂不接入真实支付交易流程，全部支付环节仅做模拟假支付交互，无真实资金流转逻辑**。整体采用BS网页架构开发，代码/页面结构需预留无缝迁移至微信小程序的扩展能力，地图能力预留高德地图全局APIKEY变量位。

我要的是手机端的，没有网页PC端！

## 一、平台参与方角色（滴滴同款生态分层，适配摩的场景）
1. C端乘客用户：普通打车下单用户
2. B端摩的司机：持证摩托车营运司机，接单、行程、结算主体
3. 平台后台管理员：运营、订单、司机、用户、风控、账单管理后台角色
4. 平台财务对账角色：单独财务权限，仅处理流水、提现、对账

## 二、产品迭代分期规划（分三期，严格划分需求优先级，一期为MVP最小上线演示版本）
### 一期：核心MVP演示能力（本次原型仅开发一期内容，满足基础演示跑通全流程）
定位：可完整演示乘客-司机订单闭环，无真实支付、无复杂运营配置，仅保留跑通打车主线流程刚需功能
1. 账号基础登录：乘客/司机手机号验证码登录、基础信息修改、退出登录
2. 高德地图基础能力：定位、起点/目的地选择、路线渲染，全局预留window.GAODE_API_KEY
3. 完整模拟打车主线：下单呼叫司机、司机接单、接驾导航、行程中实时位置、行程结束
4. 模拟假支付流程：仅展示支付弹窗、加载动画、支付成功页，不对接任何支付渠道，无真实扣款、退款逻辑
5. 基础订单查看：进行中/已完成/已取消订单列表、订单基础详情
6. 司机基础工作台：上线/下线、接单推送、行程启停操作、简单收入统计（虚拟金额）
7. 后台基础看板：实时订单概览、乘客/司机基础列表查看、简易计价参数配置

### 二期：基础运营能力（一期演示稳定后迭代，完善平台基础管理与用户体验）
定位：补齐平台基础运营、风控、用户服务能力，仍不开放真实资金提现
1. 用户侧：司机星级评价、文字评价、常用地址收藏、基础优惠券功能
2. 司机侧：资质上传+线上审核流程、乘客来电联系、历史评价查看、虚拟收入明细拆分
3. 后台运营：司机资质审核模块、订单筛选/检索、简单异常订单标记、基础违规记录登记
4. 消息通知：站内订单推送通知、审核结果通知弹窗
5. 地图优化：路线预估时长/距离、定位偏移容错处理

### 三期：平台完善商业化能力（正式上线商用版本，接入真实资金链路）
定位：完整商业化闭环，支持真实支付、提现、财务对账、精细化风控
1. 支付商业化：对接真实支付渠道，支持退款、支付失败重试、订单发票申请
2. 司机财务：银行卡绑定、真实提现申请、提现审核、提现流水、平台抽成自动计算
3. 完整财务后台：全平台收支流水导出、对账报表、平台分润配置
4. 进阶风控：恶意下单/频繁取消拦截、司机离线异常预警、订单异常自动判定
5. 增值基础功能：预约打车、客服在线聊天、优惠券发放后台配置、夜间加价规则精细化配置

## 三、分角色一期MVP核心功能（本次原型仅落地一期需求）
### （1）乘客端网页一期核心功能
1. 账号体系：手机号验证码登录/注册、退出登录、昵称修改
2. 地图定位能力：自动获取当前起点、手动输入/拖拽选择起点、目的地，调用高德地图渲染线路，全局变量window.GAODE_API_KEY统一管理密钥
3. 下单打车：单一摩的车型、预估费用展示、呼叫司机、接单前/接单后取消订单弹窗提示
4. 行程中实时页：司机实时位置地图、虚拟电话按钮、行程距离/时长实时刷新、紧急求助静态弹窗（仅展示，无真实报警）
5. 模拟结算支付：行程结束弹出费用面板，模拟支付加载动画，一键完成假支付，无真实资金交互
6. 订单中心：全部订单分类展示（待出行/进行中/已完成/已取消）、订单基础详情查看
7. 个人中心：虚拟钱包余额展示、客服静态页面入口

### （2）司机端网页一期核心功能
1. 司机账号：手机号登录、资质信息静态展示（无真实上传审核，二期实现）
2. 接单开关：上线接单/下线休息切换、新订单弹窗推送
3. 订单处理：接单/拒单、导航前往起点、行程开始/行程结束确认按钮
4. 行程地图：高德路线导航、乘客实时位置同步
5. 虚拟钱包：行程虚拟收入汇总、虚拟余额展示（提现功能仅静态页面，三期实现真实提现）
6. 订单记录：历史完成/取消订单列表

### （3）平台管理后台一期最小功能
1. 后台管理员登录页
2. 简易数据看板：实时订单总量、在线司机数、今日虚拟订单收入
3. 用户管理：乘客账号列表、账号查询
4. 司机管理：司机列表、在线状态查看
5. 订单管理：全平台实时订单监控、订单筛选、订单详情查看
6. 计价规则简易配置：后台设置摩的起步价、里程单价
7. 系统设置页：高德地图APIKEY全局配置输入框

## 五、技术约束要求
1. 整体架构BS网页，前端采用轻量页面逻辑、组件化设计，后续可快速迁移至微信小程序，禁止使用仅PC网页专属复杂插件；
2. 地图模块统一封装高德地图工具类，全局预留window.GAODE_API_KEY变量，支持后台配置密钥；
3. 严格遵循一期MVP演示标准，仅开发一期核心能力，二期、三期仅做需求规划不落地页面与逻辑；
4. 支付相关全部为模拟假流程，禁止编写真实支付接口、资金扣款、退款、提现接口逻辑；
5. 业务逻辑对标滴滴出行基础打车生态，全部流程适配摩托车短途营运场景，区别于四轮网约车；
6. 输出完整一期产品原型逻辑、页面跳转关系、角色权限划分、核心交互流程，可直接用于Codex生成前端页面原型+基础接口逻辑。
