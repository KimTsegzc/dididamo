# 后端接入方案说明

## 1. 目标

- 采用前后端分离结构，前端负责页面与交互，后端负责配置与地图服务代理。
- 高德地图采用 "JS API 2.0 + Web 服务 API" 混合方案。
- 地图 Web 服务密钥只保存在后端，不在前端明文暴露。

## 2. 为什么这样选

- JS API 2.0：适合移动 H5 的地图渲染、交互、路线展示，体验最接近滴滴类产品。
- Web 服务 API：适合后端封装搜索、地理编码、路径规划等能力，便于做统一鉴权、限流和日志。
- 兼容后续小程序：后端接口可复用，前端地图层可替换为小程序插件。

## 3. 推荐架构

### 前端职责（frontend）

- 展示地图和业务页面。
- 只调用本系统后端接口，例如：
  - GET /api/config/public
  - GET /api/map/tips
  - GET /api/map/geocode
  - GET /api/map/regeo
  - GET /api/map/route
- 不保存和展示高德 Web 服务 key。

### 后端职责（server.js + backend/config）

- 托管静态资源目录 frontend。
- 暴露系统 API：
  - GET /api/health：健康检查
  - GET /api/config/public：返回前端可见配置（仅状态/掩码）
  - POST /api/admin/config/amap-key：管理员写入地图 key
- 后续新增地图代理 API（建议）：
  - /api/map/tips -> 高德输入提示 API
  - /api/map/geocode -> 地理编码 API
  - /api/map/regeo -> 逆地理编码 API
  - /api/map/route -> 路径规划 API

## 4. 密钥与安全规范

- backend/config/runtime.json 仅保存后端配置。
- 生产环境建议改为环境变量（例如 AMAP_WEB_SERVICE_KEY）。
- 前端仅读取配置状态，不返回明文 key。
- 建议加上：
  - 管理接口鉴权（至少 token）
  - 请求频控与日志审计
  - 异常报警

## 5. 目录规范（当前）

- frontend：前端页面与静态资源
- backend：后端配置与后续服务代码
- start：启动/部署脚本集合
- server.js：当前后端服务入口

## 6. 启动方式

- 本地或 ECS：
  - bash start/restart.sh
  - bash start/status.sh（或 launcher.sh status）
- 访问地址：
  - http://<host>:4173

## 7. 高德落地顺序

1. 申请并配置 JS API 2.0 key（域名白名单）。
2. 申请 Web 服务 key（仅后端使用）。
3. 先打通 /api/map/tips、/api/map/geocode、/api/map/route。
4. 前端逐步替换地图占位为真实地图组件与路线渲染。
5. 增加监控、限流、鉴权后再放量。
